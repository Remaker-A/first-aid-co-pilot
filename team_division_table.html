FirstAid Copilot 五人分工表

Edge AI / CPR MVP

按产品运行链路拆分：感知输入 → 急救决策 → Android 交互 → 评测工具链 → 医学与交付。 原则是每个人都有清晰产物，减少 Gemma、Android、后端之间的职责重叠。

角色 具体任务与交付边界
1. 队长 / 产品 / 医学合规 / 路演 负责让项目不跑偏、讲得可信、交得完整
  • 锁定 MVP：只做成人疑似心脏骤停 CPR 第一响应，不扩散到多急救场景。
  • 医学流程：整理 AHA / 国内 CPR 指南、禁忌提示、免责声明和安全边界。
  • 外部背书：对接急诊医生、红十字会、急救培训老师，拿反馈或顾问意见。
  • 叙事交付：负责 Demo 剧本、路演稿、答辩 Q&A、技术报告主线和 README。
  • 项目管理：拆任务、控进度、每天验收关键链路，保证最终提交物齐全。
边界：不独占 Prompt 工程；医学内容负责审核，具体 Agent 实现交给 Gemma 负责人。
2. Edge Gemma / Agent 决策引擎 负责 AI 大脑：离线推理、状态机、急救指令
  • 端侧部署：验证 Gemma 4 E2B / E4B 在 Android 真机上的加载、延迟、内存和离线推理。
  • Agent 状态机：设计初筛、呼叫 120、CPR 指导、实时纠错、交接报告的流程引擎。
  • Prompt / Guardrail：锁死急救步骤，避免模型自由生成危险医疗建议。
  • 本地知识库：整理 CPR 指南、应急话术、分支规则，供 Agent 调用。
  • 统一接口：把感知事件转成下一步指导，把 Session Log 转成 Handover 报告。
边界：不做 Android 页面；只提供 PerceptionEvent → GuidanceAction / SessionLog → HandoverReport。
3. CPR 感知识别 / 质量评分 负责眼睛和评分器：看懂动作，输出结构化事件
  • 视觉识别：识别人体姿态、手部位置、胸部区域、手臂是否伸直等粗粒度信号。
  • 节奏检测：判断按压频率、过快/过慢、中断时长，目标区间 100-120 次/分钟。
  • 质量评分：设计 0-100 分合成逻辑，覆盖手位、节奏、连续性、动作稳定性。
  • 测试素材：录制 CPR 假人视频、错误动作视频，建立可回放测试集。
  • 数据输出：向 Agent 提供 hand_position、compression_rate、interruption、confidence 等字段。
边界：不决定急救流程，不写业务后端；只产出可被 Agent 消费的结构化感知结果。
4. Android 终端 / 交互实现 负责跑起来：App、传感器、语音、震动和 UI 集成
  • App 架构：搭建 Android 工程、权限、生命周期、离线运行和模块集成框架。
  • 硬件能力：接入摄像头、麦克风、GPS、震动、TTS、录制和本地存储。
  • 急救 UI：实现大字高对比界面、质量分仪表盘、节拍器、动画和状态提示。
  • 流程集成：串联 Gemma Agent、感知识别、120 拨号、定位展示、Handover 报告。
  • 异常兜底：处理断网、权限未开、识别失败、模型加载慢等演示高风险点。
边界：可以集成模型和算法，但不负责设计医疗决策规则，也不维护感知算法本身。
5. 工具链 / QA / Demo 工程 / 设计交付 负责让它稳定、好测、好演示、好提交
  • 测试工具：做视频回放、假数据注入、离线测试脚本，帮助算法和 Agent 快速联调。
  • Demo 兜底：准备识别失败时的演示模式，确保现场能完整跑完 5 分钟故事线。
  • 指标看板：统计响应时间、模型延迟、节奏误差、识别成功率和质量分变化。
  • UI/UX 细化：检查字体、颜色、文案、可读性和紧急场景下的误触风险。
  • 最终物料：剪 Demo 视频、整理截图、准备测试清单、提交包和演示设备清单。
边界：不是“打杂位”；这是黑客松稳定性负责人，保证功能能被验证、能被展示。
协作原则：Android 负责“跑起来”，Gemma 负责“想清楚”，感知负责“看明白”，工具链负责“测稳定”，队长负责“讲可信”。 项目主链路尽量保持端侧本地运行，“后端”只作为测试、回放和 Demo 辅助，不作为核心依赖。
